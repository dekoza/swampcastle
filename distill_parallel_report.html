Parallelizing "distill" (AAAK compress)

Summary

The VaultService.distill() implementation was extended to optionally parallelize the AAAK compression step (Dialect.compress) using a ProcessPoolExecutor. The worker processes perform the CPU-bound compression per drawer; the main process performs the storage write (bulk update) to avoid passing database connections across processes.

Parallelism is opt-in and controlled by the parallel_workers parameter (function argument) or environment variable SWAMPCASTLE_DISTILL_PARALLEL=1 with optional SWAMPCASTLE_DISTILL_WORKERS. The default behaviour is unchanged (sequential) to preserve compatibility.

What changed (implementation notes)

  • Workers are started with a spawn multiprocessing context to avoid inheriting fork-unsafe storage state from the orchestrator process.
  • A module-level _init_worker(config_path) initializes one Dialect instance per worker process.
  • The orchestrator submits bounded batches, gathers completed results with wait(..., FIRST_COMPLETED), preserves drawer order, and flushes metadata updates in batches.
  • Benchmarks now use repeated fresh runs, alternating execution order, and explicit sequential/parallel parity checks instead of one-shot timing.
  • Scripts: benchmarks/distill_bench.py and benchmarks/mine_bench.py

Benchmark methodology

The original one-shot timings were too weak to justify hard speedup claims. Current benchmarks use a fairer method:

  • fresh state for every measured run,
  • alternating sequential/parallel execution order to reduce warm-cache bias,
  • multiple measured runs plus warmups,
  • parity checks that fail if sequential and parallel outputs differ.

Interpret the output as machine-specific measurements. The scripts report min/mean/median/max/stdev instead of a single anecdotal number.

Caveats & Observations

  • ProcessPoolExecutor adds serialization costs (pickling) and process startup overhead, so parallelism shines for larger per-item CPU work.
  • The code avoids passing database/collection objects into worker processes (unsafe/fork-unsafe). All storage writes occur in the main process.
  • Some storage backends (e.g. LanceDB) warn about fork-safety. When using multiprocessing with backends that rely on native threads, prefer the spawn start method or run parallel distill on systems where the storage driver is known to be fork-safe.
  • We cap worker count to 32 to avoid runaway process spawning from bad env vars.

How to run the benchmark

Distill:

uv run python benchmarks/distill_bench.py -n 1000 -w 4 -m 500 --runs 5 --warmup 1

Mine:

uv run python benchmarks/mine_bench.py --files 300 --lines-per-file 80 -w 4 --runs 5 --warmup 1

Use --json on either script for machine-readable output.
