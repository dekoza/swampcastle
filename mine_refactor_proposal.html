Refactoring the mine Command for Concurrency

Proposed Architecture Changes

To address the single-core bottleneck, the architecture should be updated to a producer-consumer model utilizing ProcessPoolExecutor.

  1. Producer (Multiprocessing): A pool of worker processes reads files, parses them, and chunks the text. This is a CPU-bound task perfectly suited for multiple cores.
  2. Queue: Workers place the chunked data into a thread-safe multiprocessing.Queue.
  3. Consumer (Main Thread): A single consumer running on the main thread reads chunks from the queue, batches them, and feeds them to the EmbeddingBuffer (which manages the actual database writes and embedding API calls).

Before vs. After Comparison

Aspect Before (Current State) After (Proposed Refactor)
Execution Flow Strictly sequential loop over all files. Producer-Consumer architecture.
CPU Utilization Locked to a single core due to the synchronous for loop and Python's Global Interpreter Lock (GIL). Saturates available cores using ProcessPoolExecutor for file I/O, parsing, and chunking.
File Reading & Chunking Happens sequentially in the main thread (blocking). Delegated to worker processes, running in parallel across multiple CPU cores.
Database/Embedding Access Interleaved with file reading, causing potential idle time for the DB while waiting for the next file to be parsed. Managed efficiently by a dedicated consumer thread reading from a queue, minimizing DB connection contention.
Speed on Large Repos Scales linearly (O(n)) with single-core performance. Very slow on large codebases. Scales proportionally to the number of available CPU cores (O(n/cores)). Significantly faster.

Implementation Considerations

When decoupling the database operations, we must ensure that the StorageFactory and Collection objects are only instantiated or accessed safely. LanceDB (the default storage) or SQLite connections often cannot be safely passed across process boundaries. By keeping the embedding generation and database writes in the main thread (consumer) and moving only the text extraction/chunking to the worker processes, we avoid complex IPC serialization issues.
