Deep Dive: SwampCastle mine Command

Performance Observation

You observed that the mine command operates predominantly on a single core and fails to saturate the CPU. This behavior is directly explained by how the command is structured in the codebase.

Why is it single-threaded?

1. Sequential Processing Loop
The core of the mine function processes files in a strict, sequential for loop without any parallelization mechanisms (like ThreadPoolExecutor or ProcessPoolExecutor). 
for i, filepath in enumerate(files, 1):
    drawers, room = process_file(
        filepath=filepath,
        ...
    )
2. Embedding Bottleneck
While it uses an EmbeddingBuffer which implies batching, the actual file reading, chunking, and preparation steps are executed sequentially for every file before being added to this buffer.
3. Global Interpreter Lock (GIL)
As a standard Python application, even if threads were used, CPU-bound operations (like parsing files or generating embeddings locally) would still be constrained to a single core by the Python GIL unless multiprocessing is explicitly employed.

Execution Flow Summary

  1. Discovery: Calls scan_project() to build a list of files sequentially.
  2. Initialization: Sets up storage connections and an EmbeddingBuffer.
  3. The Loop: Iterates through files one by one using process_file().
  4. Extraction: Optionally extracts Knowledge Graph proposals after the main loop.

Potential Improvements for CPU Saturation

  • Implement concurrent.futures.ProcessPoolExecutor to parse and chunk files across multiple cores.
  • Use asynchronous I/O (asyncio) if the embedding process involves network calls to external APIs.
  • Decouple the file reading/chunking phase from the embedding generation phase into a producer-consumer model.
