Parallelizing "distill" (AAAK compress)

Summary

The VaultService.distill() implementation was extended to optionally parallelize the AAAK compression step (Dialect.compress) using a ProcessPoolExecutor. The worker processes perform the CPU-bound compression per drawer; the main process performs the storage write (bulk update) to avoid passing database connections across processes.

Parallelism is opt-in and controlled by the parallel_workers parameter (function argument) or environment variable SWAMPCASTLE_DISTILL_PARALLEL=1 with optional SWAMPCASTLE_DISTILL_WORKERS. The default behaviour is unchanged (sequential) to preserve compatibility.

What changed (implementation notes)

  • Added a module-level worker _distill_worker(args) which constructs a Dialect instance and returns (doc_id, metadata_with_aaak).
  • Main process submits doc-level tasks to a process pool using a bounded sliding window to avoid unbounded memory from many futures.
  • Main process collects worker results into an ordered update list and performs a single bulk collection.update(...) call.
  • Safeguards: worker failures are logged and skipped; BrokenProcessPool is re-raised so catastrophic pool failures are visible; max worker count is capped (32).
  • Benchmarks script added: benchmarks/distill_bench.py

Micro-benchmarks (before vs after — sequential vs parallel)

Test setup: generated 1,000 drawers each containing a repeated paragraph (multiplier = 500) to make compression CPU-bound. Measured wall-clock time for VaultService.distill() run in the same process for the sequential path and for the parallel path with varying worker counts.

ScenarioTime (s)Notes
Sequential (baseline)~2.73 sSingle-process compression of 1000 larger docs (mult=500)
Parallel — 2 workers~1.41 s~1.93× speedup vs baseline
Parallel — 4 workers~0.73 s~3.7× speedup vs baseline
Parallel — 8 workers~0.39 s~7.0× speedup vs baseline

Runtimes were measured on the development machine. Absolute timings will vary across machines. For small documents (< 50 repetitions) the compression work is too light and process overhead dominates; parallelism is most beneficial when per-document CPU work is non-trivial.

Caveats & Observations

  • ProcessPoolExecutor adds serialization costs (pickling) and process startup overhead, so parallelism shines for larger per-item CPU work.
  • The code avoids passing database/collection objects into worker processes (unsafe/fork-unsafe). All storage writes occur in the main process.
  • Some storage backends (e.g. LanceDB) warn about fork-safety. When using multiprocessing with backends that rely on native threads, prefer the spawn start method or run parallel distill on systems where the storage driver is known to be fork-safe.
  • We cap worker count to 32 to avoid runaway process spawning from bad env vars.

How to run the benchmark

Use the included script:

uv run python benchmarks/distill_bench.py -n 1000 -w 4 -m 500

Flags: -n number of drawers, -w worker count, -m repetition multiplier per document to tune CPU work.
